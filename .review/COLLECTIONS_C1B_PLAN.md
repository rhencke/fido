1. BINDING COLLECTION LAW
===============================================================================

Install this law in CLAUDE.md and ARCHITECTURE.md, and reference it from the Source Forest master plan and
PAINFUL_LESSONS.md.

1.1 Ordered sequence

Use the standard Rocq `list` when order, repetition, or positional structure is semantic.

Examples:
- declarations in one source file;
- statements in one body;
- function parameters/results as written;
- call/println arguments;
- struct fields as written;
- interface elements as written;
- source imports as written;
- runtime trace order;
- canonical source-order children;
- rollback stacks where reverse creation order matters;
- a transport/API enumeration whose identity authority is elsewhere.

1.2 Identity-keyed finite collection

Use a mature standard finite map.

Examples:
- FilePath -> GoSourceFile;
- package directory -> PackageSummary;
- NodeRef/NodeKey -> compiler fact;
- identifier -> binding bucket;
- method name -> method bucket;
- graph vertex -> adjacency set.

1.3 Membership-only unordered finite collection

Use a mature standard finite set.

Examples:
- visited vertices;
- package dependencies;
- desired filesystem targets;
- abandoned temporary paths;
- audit-root identities;
- option/fault flags where only membership matters.

1.4 Duplicate-invalid source collection

Do not let standard map `add` overwrite and erase source evidence.

Use:
- source AST sequence to preserve every occurrence;
- a standard map from key to a duplicate-preserving value/bucket where collection is semantic;
- or a duplicate-rejecting builder that checks `mem` before `add` where duplicate input makes the structural
  container invalid.

Never silently overwrite.

1.5 Graph

Use:
  standard map from vertex identity to standard finite set of adjacent vertex identities.

Do not write a custom graph container or repeated list-search graph when this shape fits.

1.6 Derived enumeration

A map/set `elements` or `bindings` list is derived.

It may be used for:
- canonical output order;
- structural recursion accepted by Rocq;
- an external API requiring a list;
- proofs about finite enumeration.

It is never a second identity authority.

1.7 Failure

A failed collection builder stays failed.

Forbidden:

  match build_collection xs with
  | Some c => c
  | None   => empty_collection
  end

unless the semantics explicitly define failure as empty, which no current Fido source/program builder does.

2. WHAT COUNTS AS A PERMITTED WRAPPER
===============================================================================

A project-specific wrapper is permitted and encouraged when it gives a domain boundary without implementing a
collection.

Allowed wrapper behavior:

- instantiate a standard functor with a domain key:
    FMapAVL.Make FilePath_OT
    MSetAVL.Make ...
    Map.Make(String)
    Set.Make(String)

- define aliases:
    GoFileMap := FileMapBase.t GoSourceFile

- rename/delegate operations:
    find_file := FileMapBase.find
    file_mem := FileMapBase.mem
    file_bindings := FileMapBase.elements

- hide irrelevant standard operations;

- enforce stronger DOMAIN construction:
    filemap_of_nodes checks duplicate paths before standard add;

- define domain folds:
    package_summaries folds source files into a standard PackageMap;

- prove project-specific semantic facts about the standard collection;

- expose opaque/sealed module interfaces while storing the standard map/set internally.

Forbidden wrapper behavior:

- define a recursive map/set/tree/trie storage datatype;

- retain a parallel association list as backing or cache authority;

- reimplement generic find/mem/add/remove/balance/union/intersection;

- use list + NoDup as the collection's public storage;

- define a generic custom multimap when a standard map with bucket values fits;

- define a generic custom graph when map-to-set fits;

- make raw standard tree constructors part of Fido's public API;

- maintain two collection implementations behind one interface;

- call a custom storage implementation a “wrapper.”

NodeTable is acceptable only because it delegates its type and operations directly to
FMapPositive.PositiveMap and contains no Fido-authored storage tree.

If no standard collection is a good fit, Claude MUST:

1. document the exact semantic mismatch;
2. identify the standard alternatives considered;
3. report an ARCHITECTURAL CONFLICT;
4. notify Rob and stop.

Claude must not autonomously implement a collection.
